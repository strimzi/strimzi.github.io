Contribute to Strimzi

Contributing to an open-source project is not just about writing code. There are many different ways you can contribute to Strimzi.

Join the community

You can start by joining the community. We have a #strimzi channel on the CNCF Slack workspace which you can join and meet other Strimzi users. We also have user and developer mailing lists. You can also come to our community calls which happen every two weeks. On the community calls, you can learn about the current work or future plans. Don’t forget that in all these places you should follow the code of conduct and just be nice to others!

Spread the word

The easiest way you can help Strimzi is by spreading the word. It can start by something simple: star Strimzi repositories on GitHub, subscribe to our YouTube channel or follow us on Twitter and like / retweet our tweets. You can of course also talk about Strimzi on your social media accounts to help more people learn about our project. If you use Strimzi in production, you can add your company to our list of adopters. And you can also add your logo to the Strimzi website. The adopters list helps other users to see who already uses Strimzi and might make them more comfortable with using it as well.

Another way to spread the word about Strimzi is by writing blog posts. If you have some interesting topics. For example about how you use Strimzi and your Strimzi use cases or about using Strimzi with other projects and technologies. You can write a blog post for our own blog which is on the Strimzi website. But you can also publish it on your personal or company blog and let us know and we can help you share it with the Strimzi community.

Talking at conferences or meetups is another great way to spread the word about Strimzi. And it doesn’t matter whether it is a small local meetup or a big conference for thousands of attendees. Let us know about your talks and we can share them with the community. If you want, we can also help to review your materials or conference proposals. If you are an organizer of a conference or meetup and want to have someone to talk about Strimzi, we can also try to help you find the right speaker.

Ask for Help or Help Others

If you are new to Strimzi, you might have many questions and or run into some beginner issues. We hope that our quickstarts, our documentation or the tutorials on our YouTube channels make it easy to get started. But don’t be afraid to ask if you have any questions or issues. The #strimzi Slack channel, the user mailing list or the GitHub discussions are the best places to ask questions or ask for help.

To get some useful and fast help, it is always good to cover the following points:

  • Check the documentation and discussion history before asking new question.
  • Explain what Strimzi and Kubernetes version you are using and in which environment you are running it. There might be known issues or incompatibilities.
  • Include as many logs and configurations as possible! A single log line or exception or just a small snippet of your custom resource might not be enough to understand the problem and to help. Ideally either share them directly in your question / issue formatted as code or as an attachment. Keep in mind that people might not be eager to help if they have to decipher the logs from an upside-down picture of your screen which you took with your smartphone. The easier you make it, the more likely it is someone will help you. To collect the available logs and configurations, you can use the report tool and just share the ZIP archive it generates.

Remember that others are helping you for free. They will try their best to help you, but they might be busy with some other work or might live in a different timezone. So be patient.

As a new user, you are always best positioned to suggest improvements to the documentation. If you misunderstood something or if you think something is not sufficiently covered, you can report it to use either on any of our communication channels or by opening an issue. And you can of course also just open a PR and fix the documentation.

If you already have some experience with Apache Kafka or with Strimzi and Kubernetes, you can try to help others by answering questions or providing guidance and suggestions. You can also help by reviewing the documentation and either suggesting or contributing improvements. You can also help by making tutorials - for example as part of the documentation, a blog post or as a video.

Our documentation is currently in English. But not everyone speaks English perfectly. So you can also help by translating our documentation or blog posts into other languages.

Provide Feedback and Improve Quality

Feedback is important for. Reading about how great Strimzi is always feels great. But also negative feedback is important and helps us make Strimzi better. You can share your feedback for example on Slack or mailing lists.

But feedback can have many different forms. And one of the ways to provide feedback is for example also reporting bugs, suggesting improvements or opening enhancements requests for features you would like to see. You can also write proposals describing new features and how they should work and be implemented. You can do it even if you cannot or do not plan to implement them - someone else might.

You can also help us to improve the quality of our releases and help us test Strimzi. As most modern projects, we rely heavily on automated tests. You can always help us improve the tests - you can add new tests to improve the test coverage, improve their execution time and so on. ut you can also help by manually testing Strimzi in different configurations, environments and in different use-cases and report the results and any issues you found.

Every Strimzi release starts with a release candidate. Testing the RCs in your environments with your configurations and making sure everything works before the final release is done is also very helpful. Fixing any bugs before the release is always better and easier than doing patch releases.

Write code

Since Strimzi is a software project, you can of course also contribute by writing code. One of the things which makes Strimzi different from most other cloud native projects is that most of its parts are written in Java. So if you want to contribute to an exciting project and work with modern technologies such as Kubernetes but do not want to learn Golang, then Strimzi is the right project for you. And if you are more interested in Apache Kafka than in Kubernetes, you will find Strimzi interesting as well.

Our developer guide should help you to learn how to develop and build the code. If you are using Strimzi, you can contribute features or improvements which you are missing. You can also check the open issues on GitHub and fix some bugs or implement some enhancements. We always try to have some issues labeled as good start which are suitable for beginners. If you plan to work on some bigger contribution, it might be good idea to check with us first - for example in the #strimzi-dev channel on CNCF Slack. Bigger changes with major impact on the APIs might require writing a proposal and having it approved by the community. For small changes and bug fixes, you can just open the PR and we will review it. Programming also doesn’t always mean just new features or bugfixes. You can also contribute to our tests for example by writing new system tests.

The main part of Strimzi are the Kubernetes operators. But don’t forget that Strimzi also has many smaller sub-projects such as for example the Bridge, the OAuth library or the Canary which is the only sub-project written in Golang instead of Java. Some of the sub-projects can be also used independently of Strimzi with any Apache Kafka installation. So even if you are not running Apache Kafka on Kubernetes, you can still find some project to contribute to.
